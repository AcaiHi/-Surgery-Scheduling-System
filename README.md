# 手術排程系統 Surgery Scheduling System

醫院手術排程管理系統，提供手術資料管理、自動排程最佳化、甘特圖視覺化與班表發布等功能。系統採用 React 前端 + Spring Boot 後端 + 獨立排程演算法的三層式架構。

## 系統概述

本系統供醫院管理每日手術房的排程作業，涵蓋以下核心能力：

- **手術資料管理**：新增、修改、刪除手術，支援 CSV 批次匯入
- **自動排程最佳化**：透過模擬退火演算法（Simulated Annealing）自動分配手術至各手術房
- **甘特圖視覺化**：以互動式甘特圖呈現手術房排程，支援拖拉調整
- **班表發布與查閱**：管理者確認排程後發布為靜態快照，供全院人員查閱
- **手術群組化**：將多台相關手術合併為一個操作單元
- **房間釘選（鎖定）**：將特定手術房排程固定，排程演算法不調動該房

## 角色與權限

系統分為三個角色，權限由低至高：

| 代號 | 角色 | 主要職責 |
|------|------|----------|
| 1 | **查看者**（外科醫師）| 查看今日班表與甘特圖 |
| 2 | **編輯者**（排程管理員）| 操作手術資料、執行排程、調整甘特圖 |
| 3 | **管理者**（系統管理員）| 編輯者所有功能 + 手術房 / 科別 / 帳號等主資料維護 |

### 登入頁

使用者以帳號密碼進行身份驗證，登入後依角色導向首頁。

**忘記密碼流程**：輸入帳號 + Email → 系統寄送 6 位數驗證碼（10 分鐘有效）→ 驗證通過後進行密碼變更。驗證碼錯誤超過 5 次則鎖定。

### 今日 / 明日班表頁 (`/system/main`)

顯示已確認發布的**靜態班表快照**。此頁面資料不隨即時異動變化，提供穩定可查閱的當日行程。

- 以甘特圖顯示各手術房當日排程
- 今日 / 明日切換
- 手術條依結束時間上色（正常綠 / 加班黃 / 超限紅）
- 支援篩選（主治醫師、手術房、科別）
- 可啟用拖拉調整快照內手術順序（僅修改快照，不寫入資料庫）
- 匯出 PDF
- **確認班表**（編輯者、管理者）— 從資料庫讀取明日手術並產生 / 更新明日快照

### 首頁甘特 (`/system/mains`)

顯示今日手術房的**即時資料庫排程**，讓醫師與管理員隨時查閱最新狀態。

- 預設為**唯讀模式**
- 管理者可啟用**編輯模式**，支援拖拉同房重排與跨房移動
- 9 維度篩選（科別、手術名稱、主刀醫師、手術室、預估時間、麻醉方式、手術原因、特殊需求、申請人）
- 匯出 PDF

### 排程管理 (`/system/shift-mgr`)

規劃**下一日（或未來）手術排程**的核心操作頁面，分為兩個頁籤：

#### 頁籤一：手術排程甘特圖

預設為**完整編輯模式**，支援：

- 拖拉手術（同房重排 / 跨房移動）
- 群組 / 解除群組手術
- 鎖定 / 解鎖手術房（鎖定後演算法不調動該房）
- 9 維度篩選（篩選啟用時拖拉自動停用）
- 點擊手術條查看詳細資訊彈窗
- **開始排程** — 執行自動排程演算法（顯示全螢幕載入動畫）
- 演算法完成後甘特圖自動重新載入

#### 頁籤二：參數設定

| 參數 | 說明 | 限制 |
|------|------|------|
| 每日手術開始時間 | 排程起始時間（如 08:30）| 須為 00/15/30/45 分整數倍 |
| 正常班結束時間 | 正常班最大時數 | HH:mm |
| 加班結束時間 | 加班最大附加時數 | HH:mm |
| 手術房清潔時間 | 兩台手術間的銜接分鐘數 | 正整數 |

另可切換各手術房的開放 / 關閉狀態（房內有手術則無法關閉）。

### 手術資料管理 (`/system/surgery-mgr`)

### 手術房管理 (`/system/OR-mgr`，管理者)

- 新增 / 修改 / 刪除手術房（支援批次操作）
- 切換手術房開放 / 關閉狀態

**手術房欄位**：房間 ID、房間名稱、房間類型（普通房 / 鉛牆房）、所屬科別

### 後端架構

標準 **Controller → Service → Repository** 模式，套件位於 `com.backend.project`。

| Controller | 路徑前綴 | 職責 |
|------------|----------|------|
| AlgorithmController | `/api/system/algorithm/` | 執行演算法、時間設定、房間釘選 |
| SurgeryController | `/api/system/surgery/` | 手術 CRUD、拖拉排序、群組化、CSV 匯入 |
| OperatingRoomController | `/api/system/operating-room(s)/` | 手術房 CRUD、狀態切換 |
| ChiefSurgeonController | `/api/system/chief-surgeon/` | 主治醫師 CRUD |
| DepartmentController | `/api/system/department/` | 科別 CRUD |
| UserController | `/api/login/`, `/api/system/user/` | 身份驗證、密碼重設、帳號 CRUD |
| ScheduleSnapshotController | `/api/system/schedule/` | 每日班表快照 |

### 前端架構

- **React 18** + react-router-dom v7
- **Vite** 建置工具
- **TailwindCSS 4** 樣式框架
- **axios** HTTP 請求

**關鍵元件**：
- `MainWrapper.jsx` — 中央集線器，持有共享狀態，包裝 DragDropContext
- `MainGantt.jsx` — 甘特圖主元件，支援拖拉手術重新定位
- `dragEndHandler.jsx` — 處理拖拉結束邏輯：重新計算時間、呼叫 API 更新
- `ORSMButton.jsx` — 觸發排程演算法執行
- `ganttData.jsx` — 取得房間與手術資料，同步釘選狀態

### 時間格式

所有時間以**從午夜起算的分鐘數**儲存（如 `510 = 08:30`）。顯示轉換在 `timeUtils.jsx` 處理。

## 安裝與啟動

### 1. 資料庫設定

```sql
CREATE DATABASE yourdatabase CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

### 2. 後端設定

編輯 `SERVER/server/src/main/resources/application.properties`，填入資料庫與郵件設定：

```properties
spring.datasource.url=jdbc:mysql://localhost:3306/yourdatabase
spring.datasource.username=your_username
spring.datasource.password=your_password

spring.mail.username=your_email@gmail.com
spring.mail.password=your_app_password
```

### 3. 啟動後端

**方式一：使用 JAR（推薦）**

```bash
java -jar server.jar
```

**方式二：使用 Maven**

```bash
cd SERVER/server
./mvnw spring-boot:run
```

後端預設啟動於 `http://localhost:8080`。

### 4. 啟動前端

**方式一：使用已建置的靜態檔**

將 `frontend/` 資料夾部署至任何靜態檔案伺服器（如 Nginx）。

**方式二：開發模式**

```bash
npm install
npm run dev
```

前端開發伺服器預設啟動於 `http://localhost:5173`。

### 5. API 位址設定

前端 API 位址設定於 `src/config.js`：

```javascript
export const BASE_URL =
  window.location.hostname === "localhost"
    ? "http://localhost:8080"
    : "http://your-server-ip:8080";
```

## 自動排程演算法

演算法為**獨立執行的 Java 子程序**，系統透過 CSV 檔案與其溝通。演算法模組位於 `SERVER/server/algorithm/`，詳細的介面規格請參閱 [`algorithm/algorithm.md`](SERVER/server/algorithm/algorithm.md)。

### 執行流程

```
管理者 / 編輯者點擊「開始排程」
       ↓
[全螢幕載入動畫，禁止其他操作]
       ↓
系統匯出三個輸入 CSV（TimeTable.csv, room.csv, Arguments4Exec.csv）
  - 排除：關閉的房間、鎖定的房間、群組從屬手術
       ↓
啟動演算法子程序（ProcessBuilder），等待完成
       ↓
後端後處理：
  1. 移除空佔位手術及其清潔行
  2. 補入鎖定房間的原始排程
  3. 補入群組從屬手術
  4. 解析結果 → 更新資料庫（手術所屬房間、房內排序）
  5. 重新計算全院優先序列
       ↓
「排程完成」→ 確認 → 重新載入甘特圖
```

### 輸入 CSV

| 檔案 | 用途 |
|------|------|
| `TimeTable.csv` | 所有待排程手術的時間表 |
| `room.csv` | 參與排程的手術房清單（含特殊房型標記）|
| `Arguments4Exec.csv` | 時間參數（開始時間、正常班時數、加班時數、清潔時間）|

### 輸出 CSV

| 檔案 | 用途 |
|------|------|
| `Guidelines.csv` | 人可讀的排程結果（後端從此解析更新資料庫）|
| `newTimeTable.csv` | 機器可讀版本（備份用）|

## 核心功能規格

### 甘特圖通用規格

```
┌──────┬────────────────────────────────────────────────────────────┐
│      │  08:00   09:00   10:00   11:00   12:00   13:00   14:00    │
├──────┼────────────────────────────────────────────────────────────┤
│  A1  │ [骨科手術 90min 🟢] [清潔 45min] [骨科手術 120min 🟡]     │
│ 鉛牆 │                                                            │
├──────┼────────────────────────────────────────────────────────────┤
│  A2  │ [普外科 60min 🟢] [清潔 45min] [普外科 180min 🔴]         │
│ 普通 │                                                            │
├──────┼────────────────────────────────────────────────────────────┤
│  A3  │ [群組：骨科+骨科 210min] [清潔 45min]                      │
│  🔒  │                                                            │
└──────┴────────────────────────────────────────────────────────────┘

🟢 正常班內結束   🟡 加班時間內結束   🔴 超過加班上限   🔒 已鎖定房間
```

**甘特圖篩選（9 維度）**：科別、手術名稱、主刀醫師、手術室、預估時間（數值範圍 + 雙滑桿）、麻醉方式、手術原因、特殊需求、申請人。篩選啟用時不符合條件的手術灰化顯示，拖拉功能自動停用。

**手術詳細資訊彈窗**：點擊手術色條開啟，顯示申請書號、病歷號、病患姓名、手術日期、科別、手術房（含類型）、手術名稱、主刀醫師、預估時間、開始 / 結束時間、麻醉方式、手術原因、特殊需求、申請人。群組手術可用箭頭瀏覽各成員。

---

### 手術群組化

多台手術合併為一個整體操作單元，適用於同一病患的連續手術或需連貫進行的多個術式。

```
群組化前：
  [手術 A 90min] [清潔] [手術 B 60min] [清潔] [手術 C 45min]
       各自獨立

群組化後：
  [══════ 群組 A+B+C  225min（含銜接清潔）══════]
       視為單一區塊
```

| 操作 | 行為 |
|------|------|
| 拖拉 | 拖拉主手術 → 所有成員跟著移動到同一房間 |
| 刪除 | 刪除任一成員 → 整個群組全部刪除 |
| 演算法輸入 | 只寫入主手術；從屬手術不列入 CSV |
| 演算法輸出 | 主手術分配到哪個房間，所有成員同步更新 |
| 解散 | 各成員回到獨立手術，依序插入原始位置 |

---

### 房間釘選（鎖定）

將特定手術房的排程固定，使排程演算法不重新分配該房間內的手術。

```
釘選前（演算法會調動 A1、A2、A3）：
  A1 [手術1][手術2][手術3]  ← 演算法可重排
  A2 [手術4][手術5]         ← 演算法可重排

釘選 A1 後（演算法只調動 A2、A3）：
  A1 🔒[手術1][手術2][手術3] ← 固定不動
  A2 [最佳化後結果]

演算法完成後 A1 的原始排程自動補回結果中。
```

- 釘選狀態存於伺服器記憶體（系統重啟後需重新設定）
- 釘選房間不列入演算法輸入
- 釘選房間不可拖入或拖出手術

---

### 手術優先序列

全院所有手術依**預估時間由長至短**自動排序，最長手術排序第 1。觸發時機：新增、刪除、修改預估時間、演算法完成後。

### 手術房型限制

特殊房型需求為 `Y` 的手術只能安排在**鉛牆房**（特殊手術房），拖拉時系統拒絕移入普通房。

### 快照機制

班表快照是靜態 JSON 檔，需管理者或編輯者手動點擊「確認班表」才產生。今日/明日班表頁讀取快照，不受即時資料異動影響，確保展示穩定。

### CSV 批次匯入規格

逐行驗證，部分失敗不中止整批，系統回傳失敗清單：

| 驗證項目 | 不通過時 |
|----------|----------|
| 申請書號已存在 | 跳過，記入失敗清單 |
| 手術房名稱不存在 | 跳過 |
| 手術房科別與 CSV 不符 | 跳過 |
| 主刀醫師姓名不存在 | 跳過 |
| 預估時間非有效數字 | 跳過 |
